# CLAUDE.md

Guidance for Claude Code working in this repository.

## What this is

**ranbits.com** — a spec-grounded resource that teaches **6G from first
principles** and tracks 3GPP as 6G is standardized. 6G is the primary focus; the
original **5G NR course (module0–12)** is preserved as a secondary
**"Foundations"** layer (same URLs).

- Pure **static site**, served by **GitHub Pages** from the **`master`** branch
  (which is the default branch). `CNAME` → `ranbits.com`. There is no other
  branch — deploy = commit to `master`.
- **Light theme.** Hand-written HTML, shared `css/main.css` + `js/main.js`. No
  framework. **No runtime dependencies.** Node (built-ins only, ≥18) is used for
  build-time scripts and the GitHub Action.

## ⚠️ Non-negotiable accuracy guardrails (read before editing 6G content)

There are **no normative 6G specifications yet** (first expected in 3GPP
Release 21, ~2029). So the site protects its credibility by rule:

- Every 6G claim is labelled exactly one of:
  - `requirement` — an agreed use case/scenario/target (ITU-R IMT-2030,
    TR 22.870, TR 38.914). **Must** carry a source + date.
  - `candidate` — a technology option under study (waveforms, sub-THz,
    AI-native, RIS, ISAC). Shown neutrally. **Must** carry a source + date.
  - `foundation` — an established 5G/earlier fundamental, taught concretely.
- **Never invent** spec numbers, document IDs, completion %, dates, or WG
  decisions. No source → mark unverified and omit the value; never guess.
- Worked numbers must be real + labelled `framework example target`,
  `study assumption`, or `5G baseline` — **never** a 6G "spec value".
- Source priority: `3gpp.org` → ITU-R IMT-2030 → reputable secondary
  (Qualcomm/Ericsson/Nokia blogs, arXiv) **clearly labelled secondary**.
- Note: `3gpp.org` returns HTTP 403 to automated fetching; cite it as canonical
  but cross-check content via search/secondary sources.

In HTML, claims use `.claim.{req|cand|found}` blocks with a `.clabel` chip and a
`.src` line. See any page in `6g/` for the pattern. `BUILD-REPORT.md` is the
full source ledger of every published fact.

## Workflow

```bash
npm run build     # render tracker dashboard, per-domain boxes, feed.xml, sitemap.xml
npm run fetch     # dry-run the daily updater (no writes); inspect scripts/.fetch-report.md
npm run update    # fetch --write + build  (what the daily Action runs)
```

- The tracker is **data-driven**: edit `data/6g-tracker.json`, then
  `npm run build`, then commit the regenerated output. Per-domain tracker boxes
  are injected into `6g/*.html` between `<!-- TRACKER:<id> -->` markers — don't
  hand-edit inside those markers.
- The daily updater (`scripts/fetch-3gpp-status.js`) is **safe by construction**:
  validates against the schema, only advances values monotonically, never
  overwrites a sourced value with an unsourced one, never crashes, and logs
  anything skipped. Don't weaken these properties.
- **After every file edit: commit and push immediately.** deploy = commit to
  master (GitHub Pages), so every change must go live at once. No batching, no
  stopping at commit:
  ```bash
  git add <files>
  git commit -m "..."
  git push origin master
  ```

## Page/markup conventions (match the existing code)

- Page skeleton: `<body><div class="page-body"> … </div>`. The sidebar is
  injected by `buildSidebar('<filename>.html')` (called at the end of each page).
  Sidebar structure (in `js/main.js`):
  - **Home + 6G Tracker** — plain links, no group label.
  - **6G Topics** — labeled group; each item prefixed with a `01`–`09` monospace
    number tag. Active item gets `aria-current="page"`.
  - **Foundations — 5G NR** — teal-labeled group; each module collapses to a
    single line (links to its first lesson). Only the active module expands to
    show sub-items indented at `padding-left:24px`. Completed lessons get class
    `done`; locked (not yet written) get class `locked`.
  - Section groups are separated by `border-top:1px solid var(--border)`.
- **Spec parameter tables** use `.spec-table-wrap` > `.spec-table-cap` +
  `table.spec-table`. Pattern:
  ```html
  <div class="spec-table-wrap">
    <div class="spec-table-cap">
      <span class="stc-title">Table title</span>
      <span class="stc-src">Source citation</span>
    </div>
    <table class="spec-table">
      <thead><tr><th>…</th></tr></thead>
      <tbody>
        <tr>                          <!-- white = foundation/established -->
          <td class="td-name">…</td>
          <td class="td-val">…</td>   <!-- blue bold numeric value -->
          <td class="td-desc">…</td>  <!-- prose, wraps -->
        </tr>
        <tr class="row-study">        <!-- amber = candidate/under study -->
          …
        </tr>
        <tr class="row-na">           <!-- grey = not applicable / TBD -->
          …
        </tr>
      </tbody>
    </table>
  </div>
  ```
  Every 6G page now has 1–2 spec tables with real sourced values (TS numbers,
  ITU-R refs, NIST FIPS numbers). Never invent values — see accuracy guardrails.
- **Interactive diagrams use `<canvas>` and stay DARK** (`#040610` background,
  light-on-dark colors) — a deliberate "screen" on the light page, consistent
  with the existing module diagrams. Page **chrome** is light (theme variables
  `--blue`, `--blue-l`, `--blue-m`, `--text`, `--text2`, `--text3`, `--muted`…).
  Don't introduce dark `--*-d` variables (they don't exist here).
- Each 6G page: full `<meta>`/OG/Twitter + JSON-LD (Article + FAQ where apt),
  `rel="canonical"`, feed auto-discovery, and forward/back links to the matching
  Foundations module.
- Foundations (module) pages: keep their URLs, SEO titles, and content. Each
  carries a `.foundations-banner` linking forward to a 6G topic.

## Don't break

- 5G module URLs (`moduleN/*.html`) — preserved for search ranking/inbound links.
- `ads.txt`, `CNAME` — leave untouched.
- The accuracy guardrails above — they are the site's whole value proposition.
